# Wiki vs Docs

## 1. 비교표

| 구분 | GitHub Wiki | 레포지토리 /docs |
| --- | --- | --- |
| 저장 위치 | 별도 Wiki 전용 git 저장소 | 메인 레포지토리 내부 |
| 브랜치 전략 | 사실상 없음 (단일 히스토리) | 일반 브랜치 전략 그대로 사용 |
| PR / 리뷰 | 기본적으로 불가능 | 가능 (코드와 동일한 프로세스) |
| Main 보호 정책 | 적용 불가 | 적용 가능 |
| 코드와의 결합도 | 느슨함 | 매우 높음 |
| 자동화(CI/CD) | 제한적 | 매우 용이 |
| 기여 진입 장벽 | 낮음 | 상대적으로 높음 |
| 장기 유지보수 | 문서 성격에 따라 편차 큼 | 구조적으로 안정적 |

---

## 2. GitHub Wiki 방식이 좋은 경우

### 2-1. “문서를 빨리 쓰고, 쉽게 고치고 싶을 때”
GitHub Wiki는 진입 장벽이 매우 낮습니다.\
웹 UI에서 바로 수정할 수 있고, 브랜치나 PR 개념을 몰라도 기여가 가능합니다.
- 초기 프로젝트, 아이디어 정리 단계에 적합합니다.
- 비개발자(기획, 운영, 연구자)가 주 기여자인 경우 부담이 적습니다.
- 문서가 “정제된 산출물”보다는 “공유 메모”에 가까울 때 잘 맞습니다.

다만 이 편의성은 곧 통제력 부족으로 이어질 수 있습니다.

--

### 2-2. 코드와 분리된 지식 공간이 필요할 때

Wiki는 메인 레포와 물리적으로 분리된 git 저장소입니다.\
이 때문에 코드 변경과 문서 변경을 강하게 묶고 싶지 않을 때 유리합니다.
- 개념 설명, 배경 이론, 참고 자료 모음
- 코드 릴리스와 무관하게 진화하는 문서
- “정답”보다 “맥락”이 중요한 지식

이런 성격의 문서라면 Wiki가 더 가볍게 작동합니다.

### 2-3. 단점은 명확하다
- PR 리뷰가 없어 문서 품질이 들쭉날쭉해지기 쉽습니다.
- Main 보호, 승인 프로세스 같은 거버넌스 설계가 어렵습니다.
- 시간이 지나면 “누가 책임지고 관리하는지 모르는 문서”가 쌓이기 쉽습니다.

즉, Wiki는 속도는 빠르지만 구조는 약한 선택입니다.

---

## 3. 레포지토리 /docs 방식이 좋은 경우

### 3-1. 문서를 “코드와 같은 자산”으로 볼 때

/docs 방식의 핵심은 단순합니다.\
문서도 코드와 동일한 규칙으로 관리한다는 전제입니다.
- 브랜치 전략을 그대로 적용할 수 있습니다.
- PR 리뷰를 통해 문서 변경의 타당성을 검증할 수 있습니다.
- Main 브랜치를 보호하면서, 문서 기여만 선택적으로 허용할 수 있습니다.

이 구조에서는 문서가 설명이 아니라 설계의 일부가 됩니다.

--

### 3-2. 의사결정 기록과 궁합이 좋다

ADR(Architecture Decision Record), 설계 변경 이력처럼\
“왜 이렇게 했는지”가 중요한 문서는 /docs와 궁합이 매우 좋습니다.
- 특정 커밋 시점의 코드와 문서를 정확히 대응시킬 수 있습니다.
- 나중에 판단을 되짚을 때 맥락 복원이 가능합니다.
- 책임 소재가 흐려지지 않습니다.

장기 운영 프로젝트나, 외부 감사·설명 책임이 필요한 경우 특히 유리합니다.

--

### 3-3. 자동화·확장성이 압도적이다

/docs는 CI/CD 파이프라인에 자연스럽게 들어갑니다.
- 문서 린트, 링크 깨짐 체크
- 특정 변경 시 문서 업데이트 강제
- Docusaurus, MkDocs 같은 정적 문서 사이트로 확장

즉, 문서가 사람의 선의가 아니라 시스템에 의해 유지됩니다.

--

### 3-4. 단점도 분명하다
- git/PR에 익숙하지 않은 사람에게는 진입 장벽이 있습니다.
- 즉흥적인 기록에는 다소 무겁게 느껴질 수 있습니다.
- 초반에는 “문서 쓰는 비용”이 높게 체감될 수 있습니다.

그래서 /docs는 정리된 지식을 오래 가져갈 팀에 적합합니다.

---

## 4. 실무적으로 가장 많이 쓰는 절충안

많은 팀이 다음 구조를 선택합니다.
- Wiki: 개념 메모, 아이디어, 러프한 정리
- /docs: 확정된 설계, 가이드, ADR, 운영 문서

즉,
- Wiki는 “생각하는 공간”
- /docs는 “책임지는 공간”

으로 역할을 나누는 방식입니다.

---

## 5. 선택 기준을 한 문장으로 정리하면
- 속도와 개방성이 중요하면 → GitHub Wiki
- 정합성·책임·자동화가 중요하면 → /docs
- 둘 다 필요하면 → Wiki + /docs 병행
