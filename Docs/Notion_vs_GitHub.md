# Notion vs GitHub

Agentic Workflow 관점에서의 구조적 비교

---

## 0. 전제 정리: Agentic Workflow란 무엇을 요구하는가

Agentic Workflow는 단순 자동화가 아니라,\
판단·반복·분기·되돌림이 가능한 작업 흐름을 전제로 합니다.

즉, 다음 조건이 충족되어야 합니다.
1. 상태(State)가 명확해야 한다
2. 변경(Change)이 이벤트로 기록되어야 한다
3. 되돌림(Rollback)과 분기(Branch)가 가능해야 한다
4. 사람이 개입하는 지점과, 에이전트가 개입하는 지점이 구분되어야 한다

이 기준을 놓고 보면, Notion과 GitHub는 전혀 다른 철학 위에 놓인 도구입니다.

---

## 1. 상태 관리 관점: 문서 상태 vs 시스템 상태

### Notion의 상태 모델

Notion에서 문서는 기본적으로 `“현재 상태의 뷰”`입니다.
- 가장 최신의 내용이 항상 우선됩니다.
- 과거 버전은 존재하지만, 기본 인터페이스에서 적극적으로 드러나지 않습니다.
- 문서가 어떤 의사결정 단계를 거쳐 지금 상태에 이르렀는지 구조적으로 표현하기 어렵습니다.

이 구조는 인간 중심의 읽기·정리에는 적합하지만,\
에이전트가 “지금 이 문서는 어떤 상태인가?”를 판단하기에는 불리합니다.


### GitHub의 상태 모델

GitHub에서 모든 것은 상태 변화의 연속입니다.
- 커밋은 명확한 상태 스냅샷입니다.
- 브랜치는 서로 다른 상태 세계선입니다.
- PR은 “이 상태를 채택할 것인가”에 대한 판단 요청입니다.

에이전트 관점에서 보면, GitHub는\
상태 머신(state machine)에 가까운 시스템입니다.

---

## 2. 변경 인식 관점: 덮어쓰기 vs 이벤트 스트림

### Notion: 변경은 결과만 남는다

Notion에서는 문서가 수정되면, 기본적으로 이전 상태가 덮여집니다.
- 누가 무엇을 왜 바꿨는지 맥락은 코멘트에 의존합니다.
- 변경 자체가 독립적인 객체(event)로 취급되지 않습니다.
- 에이전트가 “이번 변경은 어떤 의도를 가진 것인가”를 해석하기 어렵습니다.

즉, Notion에서 변경은 부수 정보입니다.



### GitHub: 변경 그 자체가 1급 객체다

GitHub에서는 변경이 곧 이벤트입니다.
- 커밋 메시지는 변경의 의도를 담습니다.
- diff는 판단 근거가 됩니다.
- PR 토론은 충돌과 합의를 구조적으로 남깁니다.

에이전트는 이를 기반으로 다음과 같은 판단을 할 수 있습니다.
- 이 변경은 기능 추가인가, 리팩터링인가
- 리뷰가 충분히 이루어졌는가
- 이 판단은 확정되었는가, 보류되었는가

이는 Agentic Workflow에서 결정적 차이를 만듭니다.

---

## 3. 분기와 실험: 병렬 사고가 가능한가

### Notion은 병렬 실험에 취약하다

Notion에서도 복제(copy)를 통해 실험은 할 수 있습니다.\
그러나 이는 구조적으로 다음 문제를 가집니다.
- 어떤 문서가 정본인지 애매해집니다.
- 실험 결과를 다시 병합하기 어렵습니다.
- 실험 간 차이를 기계적으로 비교하기 힘듭니다.

즉, Notion의 병렬성은 사람의 주의력에 의존합니다.



### GitHub는 병렬 사고를 전제로 설계되었다

GitHub의 브랜치는 처음부터 `“동시에 다른 판단을 해도 된다”`는 전제를 깔고 있습니다.
- 에이전트는 여러 브랜치에서 서로 다른 시나리오를 전개할 수 있습니다.
- 결과는 diff로 비교됩니다.
- 채택 여부는 PR이라는 명시적 판단으로 귀결됩니다.

이는 Agentic Workflow에서 말하는
Fan-out / Fan-in 구조와 정확히 맞물립니다.

---

## 4. 자동화 친화성: Agent가 개입할 수 있는 지점

### Notion에서의 Agent 개입

Notion API를 통해 에이전트가 할 수 있는 일은 제한적입니다.
- 문서 생성 및 수정
- 데이터베이스 필드 업데이트
- 간단한 트리거 자동화

하지만 다음은 어렵습니다.
- “이 변경이 적절한가?”에 대한 판단
- 변경 차이 기반 분석
- 정책 위반 여부 자동 차단

즉, Notion은 행위 실행에는 열려 있지만, 판단 자동화에는 닫혀 있습니다.



### GitHub에서의 Agent 개입

GitHub에서는 에이전트가 다음 역할을 맡을 수 있습니다.
- PR 리뷰 에이전트
- 변경 요약 및 영향 분석 에이전트
- 정책 검사(문서 누락, ADR 부재 등) 에이전트
- 병합 조건 충족 여부 판단 에이전트

중요한 점은,\
에이전트의 판단 결과가 워크플로우를 실제로 멈추거나 진행시킬 수 있다는 것입니다.

이는 Agent가 “조언자”가 아니라\
`워크플로우의 행위자(actor)`가 되는 구조입니다.

---

## 5. 책임과 통제: Human-in-the-loop 설계

### Notion: 암묵적 인간 통제

Notion에서는 사람이 항상 최종 판단자입니다.
- 에이전트가 개입하더라도, 결과는 참고 자료에 가깝습니다.
- 잘못된 변경을 시스템 차원에서 막기 어렵습니다.

이는 작은 팀에서는 유연하지만,\
Agent 수가 늘어날수록 통제 비용이 급격히 증가합니다.



### GitHub: 명시적 Human-in-the-loop

GitHub에서는 인간의 개입 지점이 명확합니다.
- 승인 버튼
- 병합 권한
- 보호된 브랜치 규칙

에이전트는 판단을 보조하거나,\
조건을 충족했는지 검사할 수는 있지만\
최종 통제 지점은 구조적으로 분리되어 있습니다.\
\
이는 Agentic Workflow에서 매우 중요한 안전장치입니다.

---

## 6. 정리: Agentic Workflow 관점에서의 선택 기준

| 관점 | Notion | GitHub |
| 상태 모델 | 현재 상태 중심 | 상태 변화 중심 |
| 변경 기록 | 결과 위주 | 이벤트 위주 |
| 병렬 사고 | 취약 | 구조적 지원 |
| Agent 자동화 | 실행 중심 | 판단 + 실행 |
| 통제 구조 | 암묵적 | 명시적 |
| 확장성 | 인간 스케일 | 시스템 스케일 |

---

결론을 한 문장으로 정리하면
- Notion은 인간의 사고를 정리하는 공간이고
- GitHub는 인간과 에이전트가 함께 판단을 축적하는 시스템입니다.

Agentic Workflow를\
“AI가 문서를 써주는 자동화”가 아니라\
판단과 책임이 축적되는 작업 시스템으로 본다면,\
GitHub는 선택이 아니라 전제 조건에 가까운 도구입니다.
